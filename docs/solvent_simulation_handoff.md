# Explicit Solvent Simulation Handoff

## Purpose

This document is a handoff for a separate project dedicated to short explicit-solvent simulation of CSP receptor conformers generated by `poly_csp`.

The target use case is:

- take one or more dry CSP receptor conformers from `poly_csp`
- run short explicit-solvent relaxation / MD
- extract dry receptor conformers that are more plausible in solvent
- export those dry conformers in a format that is directly interchangeable with the current docking handoff

The docking module should not need to know whether a receptor came directly from `poly_csp` or from the follow-on solvent-simulation project.


## Why This Should Be A Separate Project

This is better handled outside the core `poly_csp` repo.

Reasons:

- `poly_csp` is currently strongest as a deterministic CSP builder, ranker, and dry-receptor exporter.
- Explicit-solvent simulation introduces a different toolchain and operating model:
  - solvent packing
  - box construction
  - temperature / pressure control
  - trajectory storage
  - clustering and time-series analysis
  - HPC job management
- Organic-solvent setup is materially more complex than the current OpenMM dry-receptor workflow.
- The resulting data volume and protocol iteration cadence are much closer to an MD project than to a receptor-construction project.

Recommended project boundary:

- `poly_csp`: build, rank, and export dry CSP receptor seeds
- new project: solvate, simulate, analyze, and re-export dry docking receptors


## Engineering Guardrails

The new project should inherit the same software discipline as `poly_csp`.

These are design constraints, not preferences:

- one canonical path per supported workflow
- configuration-driven behavior through Hydra
- modular, typed components with explicit contracts
- DRY implementation with shared utilities instead of copy-pasted logic
- clean code over broad fallback behavior
- explicit failures over silent compatibility shims

Practical interpretation:

- Do not build parallel codepaths for the same task unless there is a real scientific or operational reason.
- Prefer one well-specified ingest path for `poly_csp` receptor bundles rather than multiple partially overlapping loaders.
- Prefer one canonical export path for docking-compatible dry receptors rather than format-specific special cases.
- Use Hydra configs for protocol definition, solvent choice, restraint policy, clustering settings, and export policy.
- Keep config structure orthogonal and composable rather than encoding behavior in ad hoc booleans scattered through the code.
- Avoid compatibility shims whose only purpose is to preserve old internal APIs. If the architecture changes, update call sites cleanly.
- Avoid “best effort” fallbacks that silently switch protocols, skip validation, or downgrade physics.
- Fail fast when atom order, topology identity, solvent model availability, or export compatibility is violated.
- Reuse shared export and QC logic where possible instead of reimplementing slightly different versions.
- Keep simulation setup, MD execution, analysis, and export in separate modules with narrow interfaces.

Recommended architectural shape:

- `ingest/`: strict loading and validation of `poly_csp` receptor bundles
- `setup/`: solvated-system construction only
- `simulate/`: minimization, equilibration, and MD execution only
- `analyze/`: solvent stripping, clustering, and frame selection only
- `export/`: dry receptor artifact generation only
- `qc/`: compatibility and chemistry checks only

Important anti-patterns to avoid:

- rebuilding CSP chemistry in the solvent project
- format-dependent branches that produce slightly different receptor semantics
- multiple exporter implementations for the same `pdbqt` or Vina-box product
- simulation code that also mutates export metadata or ranking logic
- permissive loaders that “fix up” broken atom mappings instead of rejecting them


## Core Design Rule

The new project must treat `poly_csp` outputs as the chemical and topological source of truth.

It must not:

- rebuild CSP chemistry from scratch
- reparameterize the dry CSP if a valid `model.prmtop` already exists
- change atom ordering in the dry receptor
- emit top-level docking products that differ in meaning or filename from the current `poly_csp` handoff

It may:

- add solvent and optional ions around the receptor
- run short restrained or partially restrained MD
- strip solvent and export selected dry receptor frames
- rank or cluster those dry receptor frames

## Required Input Bundle For The New Project

Each input receptor seed should come from either:

- a top-level `poly_csp` output directory, or
- a `ranked_XXX/` directory from a `poly_csp` multi-start run

Minimum required files:

- `model.prmtop`
- `model.inpcrd`
- `model.pdb`
- `build_report.json`

Strongly recommended:

- `model.sdf`
- `receptor.pdbqt`
- `vina_box.txt`
- `resolved_config.yaml`
- parent `ranking_summary.json` if the receptor came from a ranked run

Input assumptions:

- atom order is authoritative and must be preserved
- topology is fixed
- charges and LJ parameters in the dry receptor are authoritative
- the receptor is already all-atom and docking-capable
- periodic-origin conformers should already have been expanded to open dry receptors by `poly_csp`

The new project should consume the open receptor, not the raw periodic optimization cell.


## Scientific Goal

The scientific role of the new project is not to discover CSP order from scratch.

That has already been handled upstream by `poly_csp`.

The role here is:

- start from a chemically sensible, ranked dry receptor
- let the selector / connector region adapt in solvent
- sample short-time solvent-conditioned conformational drift
- return a small ensemble of dry receptor conformers likely to be relevant in solvent

This implies a restrained, receptor-focused workflow, not unconstrained long-timescale coating reorganization.


## Recommended Scope

### In scope

- short explicit-solvent relaxation
- short explicit-solvent MD
- neat methanol and neat isopropanol as initial solvent targets
- possibly a later mixed-solvent extension
- restrained simulations that preserve helix integrity while allowing selector adaptation
- clustering and frame selection on the dry receptor
- export of dry receptor bundles for docking

### Out of scope for the initial version

- full chromatographic mobile-phase realism
- adsorption to silica or stationary-phase support
- large-scale polymer reorganization
- direct docking against solvated systems
- mixed-solvent free-energy workflows
- chemistry rebuilding or reparameterization of the CSP core

Logical pipeline:

2. validate atom order, topology, and metadata

6. strip solvent and analyze receptor-only trajectory
7. cluster / rank dry receptor frames
8. export selected dry conformers using the existing docking contract


## Recommended Technology Choices

### Simulation engine

- OpenMM for minimization, equilibration, and MD

### System assembly

Use Amber-compatible assembly tools, not OpenMM `Modeller.addSolvent()`, for the main organic-solvent workflow.

Why:

- the CSP already arrives as `prmtop` / `inpcrd`
- the dry receptor uses a mixed GLYCAM + GAFF2-derived parameterization
- `Modeller.addSolvent()` is not a clean fit for methanol or isopropanol
- water-oriented convenience APIs do not solve the organic-solvent packing / parameter problem

Recommended assembly stack:

- AmberTools
- ParmEd
- Packmol

OpenMM remains the simulation engine after assembly.

### Analysis

- MDTraj or MDAnalysis for trajectory handling
- NumPy / SciPy for clustering and summary analysis

### Reuse from `poly_csp`

Prefer importing small pieces from `poly_csp` rather than copying them:

- PDBQT export logic
- Vina box generation logic
- QC calculations where applicable

If cross-project import becomes awkward, factor these into a tiny shared compatibility layer rather than duplicating logic loosely.


## Solvent Model Recommendations

### Initial solvent targets

Implement in this order:

1. vacuum restart / dry MD smoke test
2. neat methanol
3. neat isopropanol
4. mixed solvents later

### Forcefield compatibility

Use solvent models compatible with the Amber-family solute forcefield environment.

Do not casually mix:

- Amber-family solute parameters
- unrelated solvent forcefields

unless that combination is intentionally validated.

For methanol and isopropanol, the new project should select one Amber-compatible solvent model per solvent and treat that choice as part of the project contract.

The exact solvent model choice should be explicit in config and written into output metadata.


## Critical Engineering Constraint: Atom Order Must Stay Fixed

This is the single most important compatibility rule.

The dry receptor exported after simulation must preserve:

- atom count
- atom order
- residue numbering
- chain IDs where possible
- atom naming where possible

Why:

- `model.prmtop` can then be reused directly for the selected dry frame
- downstream docking export remains straightforward
- the docking module can swap receptor sources without special handling

If atom order changes, the handoff becomes fragile and direct compatibility is lost.


## Recommended Solvated-System Build Strategy

Use the dry receptor `model.prmtop` / `model.inpcrd` as the immutable solute definition.

Recommended solvent setup flow:

1. load dry receptor from `model.prmtop` / `model.inpcrd`
2. generate or load solvent coordinates around the receptor
3. build a solvated Amber-compatible topology with unchanged solute atom order
4. write solvated topology / coordinates for simulation

Practical note:

- `OpenMM` alone is not the hard part
- creating a correct organic-solvent solvated system is the hard part

For organic solvents, a realistic MVP is:

- predefine solvent residue templates and parameters
- use Packmol for initial solvent placement
- use ParmEd / AmberTools-based merging for the final solvated topology


## Recommended MD Protocol

The first useful protocol should be short and conservative.

### Stage A: validation / restart

- load dry receptor only
- run brief vacuum minimization or very short vacuum MD
- confirm restart, atom mapping, and export compatibility

### Stage B: solvated minimization

- add solvent
- strongly restrain receptor heavy atoms initially
- minimize solvent and bad contacts

### Stage C: restrained equilibration

- heat gently
- keep helix-core atoms strongly restrained
- optionally keep weaker restraints on the rest of the receptor
- allow solvent and selector region to adapt

### Stage D: short production MD

- reduce restraints to the intended production level
- keep a helix-preserving restraint policy if needed
- collect a short trajectory for receptor ensemble extraction

Recommended initial restraint philosophy:

- keep helix-core atoms restrained
- allow exocyclic substitution sites, connectors, and selectors to move
- avoid unrestrained large-scale receptor collapse or helix breakage

This matches the current design direction already established in `poly_csp`.


## Frame Selection Strategy

Do not select output receptors by raw total potential energy of the solvated system.

That number depends strongly on box size and solvent count and is not a useful cross-run ranking metric.

Instead:

- strip solvent and ions
- analyze receptor-only coordinates
- cluster receptor conformations
- select representatives from the most populated and chemically reasonable clusters

Useful ranking / selection criteria:

- cluster population
- receptor QC after solvent stripping
- selector aromatic planarity
- helix symmetry retention
- H-bond donor occupancy
- absence of bad steric collapse
- optionally solvent-exposure descriptors for selector motifs

Top-level meaning:

- `ranked_XXX/` directories are alternative dry receptor conformers

### Important rule

Top-level docking artifacts must be dry receptor artifacts only.

Do not place:

- solvent atoms
- ions
- solvated topologies
- solvated coordinates

into the top-level `model.pdb`, `model.inpcrd`, `model.prmtop`, or `receptor.pdbqt`.

Put solvated products elsewhere, for example:

```text
<run>/
  solvated/
  trajectory/
  analysis/
  ranked_001/
  ...
```


## How To Produce Docking-Compatible Dry Exports

For a selected dry frame:

1. strip solvent and ions
2. confirm atom order matches the input receptor exactly
3. write updated dry coordinates
4. reuse the dry receptor topology and parameters
5. regenerate docking exports

Recommended artifact generation strategy:

- `model.prmtop`: reuse the input dry receptor `prmtop` if topology is unchanged
- `model.inpcrd`: write new dry coordinates for the selected frame
- `model.pdb`: write dry coordinates with preserved atom naming / residue identity
- `receptor.pdbqt`: generate from the unchanged dry receptor charges and selected dry coordinates
- `vina_box.txt`: generate with the same central-window logic used by `poly_csp`

To guarantee compatibility, the new project should either:

- import the relevant export helpers from `poly_csp`, or
- reproduce them exactly and test byte-level or field-level compatibility

Reusing `poly_csp` helpers is preferred.

## Recommended Compatibility Checks

Every selected dry frame should pass the following before export:

1. atom count unchanged
2. atom order unchanged
3. residue identity unchanged
4. dry receptor topology unchanged
5. dry receptor charge count unchanged
6. aromatic ring planarity acceptable
7. helix-core geometry acceptable
8. no catastrophic steric collapse

If any of these fail, reject the frame for docking export.


## Suggested Development Phases

### Phase 1: dry restart compatibility

Goal:

- prove that `poly_csp` dry outputs can be ingested and re-exported without changing meaning

Deliverables:

- load `model.prmtop` / `model.inpcrd`
- write a selected dry frame back out
- reproduce compatible `model.pdb`, `receptor.pdbqt`, `vina_box.txt`, `model.prmtop`, `model.inpcrd`

### Phase 2: solvated MVP

Goal:

- run one short explicit-solvent simulation for one receptor seed

Deliverables:

- solvated system build
- restrained minimization and equilibration
- short trajectory
- one selected dry conformer exported compatibly

### Phase 3: ranked solvent-conditioned conformers

Goal:

- run multiple receptor seeds and export several solvent-conditioned dry receptors

Deliverables:

- `ranking_summary.json`
- `ranked_001/`, `ranked_002/`, ...
- top-level best receptor

### Phase 4: broader solvent support

Goal:

- support methanol and isopropanol robustly
- possibly later support mixtures


## Recommended First Inputs

Start from the current best-supported upstream inputs:

- `poly_csp` periodic multi-start runs with `periodic_handoff.enabled=true`
- dry top-level receptor bundle or `ranked_XXX/` bundles

This avoids raw periodic-cell handling and starts from the same open receptors already used for docking.


## Final Recommendation To The Next Agent

Build the new project as a strict downstream consumer of `poly_csp` receptor bundles.

Prioritize:

1. compatibility of dry outputs
2. fixed atom ordering
3. conservative restrained explicit-solvent sampling
4. clustering-based dry frame selection
5. exact preservation of the current docking interface

Do not start by building a general-purpose MD framework.

Start by proving this narrow end-to-end path:

- ingest one `poly_csp` receptor
- solvate it
- run short restrained MD
- strip solvent
- export one dry receptor bundle that the docking module can use immediately without any code changes
